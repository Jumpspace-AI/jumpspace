# Security Policy

## Supported Versions

Jumpspace is preparing its first public `0.1.0` release. Until a public package is published, security fixes target the current mainline development version.

After launch, supported versions will be documented here by release line.

## Reporting A Vulnerability

Please do not open a public issue for a suspected vulnerability.

Use GitHub private vulnerability reporting if it is enabled for the repository. If it is not enabled yet, contact the maintainer through a private channel listed on the GitHub profile and include:

- Affected version or commit.
- Steps to reproduce.
- Impact and expected behavior.
- Whether the issue affects local files, generated task metadata, CI workflows, package contents, or agent execution.

We will acknowledge valid reports as quickly as practical and publish remediation notes once a fix is available.

## Scope

Jumpspace is a local developer tool. Security-sensitive areas include:

- File mutation commands such as `bootstrap apply`, `link update`, `repair --apply`, `plan save`, `step complete`, `status`, and `verify`.
- GitHub Actions workflow behavior generated by `init --ci github`.
- Package contents included in `npm pack`.
- Agent instructions installed by `add-skill` or `init --agent`.

Jumpspace should not require secrets for local use. Do not put credentials in task blocks, generated indexes, launch trackers, or issue reports.
